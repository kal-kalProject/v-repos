# 40 — DI — análisis y refactor

> **Veredicto.** El subsistema DI está en plena transición de un modelo basado en **decoradores + `getDiRoot()` + metadata-store** a un modelo **`Injector` + `BindingRegistry` + `inject()` + `runInInjectionContext()`**. Ambos coexisten en el árbol pero el barrel `di/index.ts:62` ya marca toda la familia de decoradores `@v-ignore — pending new architecture (main() bootstrap)`. Fase 1 cierra esta migración.

Referencias canónicas: `01-cierre-decisiones-ts.md §5–§7`, `10-provider.md`, `13-agent.md`, `21-extension.md`.

---

## 2. Las dos vías

### 2.1 Vía clásica (a deprecar)

```ts
@Injectable({ lifetime: 'singleton' })
class UserService {
  @Inject(IDb) db!: Db;
}
```

- `@Injectable` registra metadatos en `injectableStore` vía `getDiRoot().key('injectable')`.
- `@Inject(token)` es un **field decorator** que guarda `{ token }` como metadato del campo.
- En tiempo de construcción, alguien (¿quién? — mira `collect-inject-declarations.ts`) recolecta los metadatos y resuelve los campos contra un Injector ambiente.
- `bootstrap(rootModule)` toma un `@vModule`-decorado, busca un `injector` colgado de él, y arranca.

**Problema.** Depende de un singleton `getDiRoot()` que mantiene estado global, dificulta testing aislado, no se puede tener dos aplicaciones en el mismo proceso, y mezcla pésimamente con HMR.

### 2.2 Vía moderna (la canónica)

```ts
@Injectable
class UserService {
  private readonly db = inject(IDb);
  private readonly cfg = inject(CONFIG_TOKEN, { optional: true });
}
```

- `@Injectable` (decorador de clase, sin opciones obligatorias) sigue existiendo pero **sólo como marcador**: registra el token en el módulo activo.
- La inyección **no es por field decorator**. Es la función `inject()` ejecutada **dentro de `runInInjectionContext`**, normalmente durante la construcción.
- `Injector` resuelve `Binding`s tipados (registrados en módulos) y aplica la `ResolutionPipeline`: `checkCache → applyActivation → injectFields → applyLifecycle → applyDeactivation → wrapResolutionError`.
- `getCurrentInjector()` y `getCurrentScope()` son la API ambiente.
- `@App({ imports, providers })` decora la clase raíz y `bootstrap()` construye el árbol.

**Ventajas.** Sin singleton global, testeable, dos AppRef pueden coexistir, alineado con Angular zoneless / signals.

## 3. Doctrina del cierre (qué consagra el cierre TS)

`01-cierre-decisiones-ts.md` exige:

- **Sólo decoradores de clase.** `@Component`, `@Directive`, `@Pipe`, `@Injectable`. Nada de `@Input`, `@Output`, `@Inject` de campo.
- **Atoms y funciones reactivas para entradas/salidas.** `input()`, `output()`, `inject()`.
- **Zoneless.** Sin `Zone.js`. La reactividad la lleva el grafo de atoms.

→ La vía moderna ya cumple. La vía clásica no. **Decisión: borrar `@Inject` campo, conservar `@Injectable` clase como marcador.**

## 4. Token system

```ts
import { InjectionToken, TokenRef } from '@vortech/core/di';

const CONFIG = new InjectionToken<AppConfig>('AppConfig');
class IDb { /* abstract */ }

inject(CONFIG);
inject(IDb);
inject(IDb, { optional: true });
```

- `ClassToken<T>` — tu clase abstracta es token (estándar moderno).
- `InjectionToken<T>` — para tipos no-clase (config objects, strings, factories).
- `TokenRef<T>` — token diferido para cycles (forward reference).

**Decisión.** Mantener los tres.

## 5. Bindings

```ts
bind(IDb).to(SqlDb).in('singleton');
bind(CONFIG).toConstant({ apiUrl: '...' });
bind(IUserRepo).toFactory(ctx => new UserRepo(ctx.get(IDb)));
bind(IFlag).toAlias(IFeatureFlag);
bind(IDb).toClass(MockDb).when(ctx => ctx.tags.includes('test'));
```

- `BindingType`: `'constant' | 'class' | 'factory' | 'async-factory' | 'alias' | 'conditional'`.
- `LifeTime`: `'singleton' | 'transient' | 'scoped' | 'request'`.
- `ActivationHook` y `DeactivationHook` — corren al crear / destruir instancias.

**Decisión.** Es la API canónica. Documentada en [`10-provider.md`](../10-provider.md).

## 6. Decisiones de Fase 1 sobre DI

| # | Decisión | Acción concreta |
|---|---|---|
| D1 | `@Injectable` como class decorator marcador (sin opciones obligatorias). Acepta opcionalmente `{ lifetime, multi, tags, name }`. | Reescribir `injectable-decorator.ts` para que use `mainStore` (de `main.ts`) en vez de `getDiRoot()`. |
| D2 | `@Inject` (field decorator) **se elimina**. | Marcar como `@deprecated`; agregar codemod. Migrar todo consumidor a `inject()` función. |
| D3 | `getDiRoot()` y `di-root.ts` **se eliminan**. | Cualquier referencia se mueve a `mainStore` del `main.ts`. |
| D4 | `bootstrap.ts` (viejo) **se elimina**. | `bootstrap()` canónico vive en `main.ts` y usa `@App`. |
| D5 | `di-context.ts`, `create-di-context.ts`, `di-context-options.ts` **se eliminan**. | Reemplazado por `injection-context.ts`. |
| D6 | `families/` (`bootstrap-host`, `http/`) **se mueve fuera de core** o se elimina. | No es responsabilidad del core conocer HTTP. Si tiene futuro, vive en `@vortech/wire` o `@vortech/runtime`. |
| D7 | `decorators/` (los 4 builders) **se evalúan**: si sólo los usan los decoradores legacy, se eliminan. Si los usan otros sistemas (extensions slots, history extensions), se conservan en `@vortech/core/builders`. | Auditar usuarios. |
| D8 | `Injector.tree: ScopeTree` queda como API pública del injector. | Sin cambios (ya está). |
| D9 | `runInInjectionContext` queda como única vía de "ambient injector". | Sin cambios. |
| D10 | `OnDispose` lifecycle queda como contrato canónico (símil a `IDisposable` .NET). | Sin cambios. Documentar en [`10-provider.md §lifecycle`](../10-provider.md). |

## 7. Lifecycle del Injector (referencia)

`Injector.dispose()` ejecuta en este orden (`injector.ts:130-148`):

1. `children.dispose()` — depth-first.
2. `_lifecycle.dispose()` — `OnDispose` + deactivation hooks de instancias activas.
3. `_registry.clear()` — limpia bindings.
4. `_cache.clearValues()` — limpia instancias resueltas.
5. `_tree.unregister(path)` — quita el nodo del scope tree.
6. `changes.emit({ type: 'disposed', path })`.
7. `scope.dispose()` — el `IAtomOwner` libera atoms del injector.
8. `hooks.destroy()` — explícito tras notificación `after:dispose`.

**Regla.** Cualquier extensión (`families/`, `extensions/`) que mantenga estado dentro de un Injector debe registrar su limpieza vía `OnDispose` o vía `injector.cleanup.add(fn)` para no fugar.

## 8. Hooks del Injector

`InjectorHooks` (`hooks/injector-hooks.ts`) expone observables:

| Hook | Cuándo se dispara |
|---|---|
| `before:registering` / `after:registered` | Al `injector.register(...binding)` |
| `before:resolving` / `after:resolved`     | Al resolver un token |
| `before:disposing` / `after:disposed`     | Al disponer un binding |
| `creating-child` / `child-created`        | Al crear un injector hijo |

Útil para devtools, logging, métricas. Mantener sin cambios.

## 9. Correspondencia con `dotnet/`

| TS | .NET | Notas |
|---|---|---|
| `Injector` | `IInjector` (`Vortech.Core.Di`) | Mismo árbol jerárquico. .NET usa interfaz. |
| `inject<T>(token)` | `Injector.Get<T>()` o `IServiceProvider.GetRequiredService<T>()` | En .NET no hay "ambient" idiomático fuera de DI inyectada por constructor. |
| `runInInjectionContext` | `using (var s = injector.AsCurrent())` | Patrón using + AsyncLocal. |
| `@Injectable` | `[Injectable]` (atributo) o convención `IXxx → Xxx` | Marcador opcional. |
| `bind(IDb).to(SqlDb).in('singleton')` | `services.AddSingleton<IDb, SqlDb>()` o `services.Bind<IDb>().To<SqlDb>().InSingletonScope()` | API fluida equivalente. |
| `OnDispose` | `IDisposable` / `IAsyncDisposable` | BCL nativo. |
| `InjectorHooks` | `IInjectorHooks` / `IObservable<InjectorEvent>` | Mismas señales. |
| `InjectionToken<T>` | `ServiceKey<T>` o `Tag<T>` | Tipo-safe key para no-clases. |
| `ResolutionPipeline` | `IResolutionPipeline` con etapas explícitas | Misma idea, fácil de portar. |

9. ⬜ Actualizar `di/index.ts` barrel: descomentar lo que sobrevive, borrar el resto.

## 11. Lo que NO se toca

- Implementación de `Injector` (`injector.ts`) — sólida.
- `BindingRegistry`, `ResolutionCache`, `ResolutionPipeline` — sólidos.
- `injection-token.ts` — completo.
- `lifecycle/`, `hooks/`, `errors/` — completos.
- `module/` — completo.

## 12. Migración de consumidores

Consumidores externos que usen `@Inject(token) field!: T` → reemplazo:

```ts
// antes
class S {
  @Inject(IDb) db!: IDb;
}

// después
class S {
  private readonly db = inject(IDb);
}
```

Codemod sugerido (a escribir en `scripts/`): TypeScript AST que encuentre el patrón y lo reescriba.

## 13. Riesgos

| Riesgo | Mitigación |
|---|---|
| Consumidores con `@Inject` campo se rompen | Codemod + período de coexistencia (`@deprecated` warn por una versión). |
| `getDiRoot()` referenciado fuera de core | Grep en monorepo antes de borrar; mantener export proxy a `mainStore` durante una versión. |
| `bootstrap()` viejo usado en producción | Renombrar a `legacyBootstrap` y marcar deprecated antes de borrar. |
| Tests acoplados a la vía clásica | Migrar tests a la vía moderna como parte del PR. |
